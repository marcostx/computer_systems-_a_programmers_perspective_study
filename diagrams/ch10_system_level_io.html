Kernel Data Structures for I/O

Three tables manage open files: per-process descriptor table, system-wide open file table, and v-node table.

Descriptor Table (per-process)

fd 0→ File A
fd 1→ File B
fd 2→ File B
fd 3→ File C

Open File Table (system-wide)

File Apos=300, refcnt=1
File Bpos=0, refcnt=2
File Cpos=1024, refcnt=1

v-node Table

File info Astat, size, type
File info Bstat, size, type
File info Cstat, size, type

Descriptor table: Each process has its own. Maps fd integers to entries in the open file table.
Open file table: Shared by all processes. Each entry has file position, reference count, and pointer to v-node.
v-node table: One per file. Contains file metadata (stat info). Shared across all open instances of the same file.

File Sharing Scenarios

Different processes can share files in various ways through the kernel data structures.

Same file, different descriptors

Two open() calls on same file create separate open file table entries with independent file positions. Writes don't interfere.

After fork()

Child inherits parent's descriptor table. Both point to SAME open file table entries. They share the file position! Parent reads, child's position advances too.

After dup2()

Two descriptors in the same process point to the same open file table entry. They share position and flags. Used for I/O redirection.

I/O Redirection with dup2()

dup2(oldfd, newfd) copies oldfd to newfd, closing newfd first if open.

Redirecting stdout to a file

// Redirect stdout to a file int fd = open("output.txt", O_WRONLY|O_CREAT, 0644); dup2(fd, 1); // fd 1 (stdout) now points to output.txt close(fd); // close the extra descriptor printf("Hello"); // writes to output.txt, not terminal!

Before dup2:

fd 0→ stdin (keyboard)
fd 1→ stdout (terminal)
fd 2→ stderr (terminal)
fd 3→ output.txt

After dup2(3, 1):

fd 0→ stdin (keyboard)
fd 1→ output.txt
fd 2→ stderr (terminal)
fd 3→ output.txt

Robust I/O (Rio) Package

Handles short counts automatically - when read/write returns fewer bytes than requested.

Causeread()write()
EOF reachedYesNo
Reading from terminalYes (line at a time)No

When to Use Which I/O

Guidelines for choosing between Unix I/O, Standard I/O, and Rio.

ScenarioRecommendedWhy
Disk files (simple)Standard I/O (fopen, fprintf)Convenient, buffered, portable
Network socketsRio (rio_readlineb, etc.)Standard I/O has restrictions on sockets (can't mix read/write without fflush/fseek)
Signal handlersUnix I/O (read, write)Only async-signal-safe I/O functions
Binary I/ORio or Unix I/OStandard I/O adds unwanted buffering/translation
File metadataUnix I/O (stat, fstat)No standard I/O equivalent

Rule: Do NOT use standard I/O on network sockets. The buffering and restrictions make it error-prone for bidirectional network communication.
