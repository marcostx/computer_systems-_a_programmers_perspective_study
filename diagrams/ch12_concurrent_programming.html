Three Approaches to Concurrency

Three fundamental mechanisms for building concurrent programs.

Process-Based

fork() a new process for each client. Each process has separate address space.

+ Clean separation, no shared state bugs
+ Simple programming model
- High overhead (fork is expensive)
- Hard to share data between processes
- IPC required for communication

I/O Multiplexing

Single process uses select()/epoll() to handle multiple connections event-driven.

+ Single address space, easy data sharing
+ Low overhead, very scalable
+ Full control over scheduling
- Complex coding (state machines)
- Cannot exploit multi-core CPUs
- Granularity of concurrency is coarse

Thread-Based

Create threads (pthreads) for each client. Threads share the same address space.

+ Easy data sharing (shared memory)
+ Lower overhead than processes
+ Exploits multi-core parallelism
- Synchronization bugs (races, deadlocks)
- Harder to debug
- Unintentional sharing is dangerous

Shared vs Private Memory

Understanding what threads share is crucial for writing correct concurrent programs.

MemoryShared?Examples
Global variablesSHARED.data and .bss segments
Heap (malloc)SHAREDDynamically allocated memory
Code (.text)SHAREDAll threads execute same code
Thread stackPRIVATELocal variables, function args
Thread IDPRIVATEEach thread has unique TID
RegistersPRIVATEEach thread has own register state

Warning: Even though stacks are "private", threads CAN access other threads' stack variables via pointers. This is a common source of bugs (e.g., passing pointer to local variable to another thread, then the local goes out of scope).

Producer-Consumer Problem

Classic synchronization problem: producers add items to a bounded buffer, consumers remove them.

Three Semaphores

SemaphoreInitial ValuePurpose
mutex1Protect buffer access (mutual exclusion)
slotsn (buffer size)Count of empty slots (producer waits if 0)
items0Count of available items (consumer waits if 0)

Producer

P(&slots); // wait for empty slot P(&mutex); // lock buffer insert_item(); V(&mutex); // unlock buffer V(&items); // announce new item

Consumer

P(&items); // wait for item P(&mutex); // lock buffer remove_item(); V(&mutex); // unlock buffer V(&slots); // announce empty slot

Deadlock

A set of threads is deadlocked when every thread is waiting for a resource held by another thread in the set.

T1
holds A
wants B →
← wants A
T2
holds B

Four Conditions for Deadlock

#ConditionDescription
1Mutual ExclusionResources cannot be shared (e.g., mutexes)
2Hold and WaitThread holds resources while waiting for more
3No PreemptionResources cannot be forcibly taken away
4Circular WaitCircular chain of threads waiting for each other

Prevention: Break condition 4 by always acquiring locks in the same order (lock ordering). If T1 and T2 both always lock A before B, deadlock is impossible.

Thread Safety

A function is thread-safe if it always produces correct results when called from multiple threads.

Four Classes of Thread-Unsafe Functions

ClassProblemFix
1. Unprotected shared variablesFunctions that read/write shared globals without synchronizationAdd mutex protection
2. State-dependent functionsFunctions that rely on persistent state across calls (e.g., rand, strtok)Use reentrant version (_r suffix) or pass state as argument
3. Functions returning static pointersFunctions like ctime, gethostbyname that return pointer to static bufferUse lock-and-copy: lock, call, copy result, unlock. Or use _r versions
4. Functions calling unsafe functionsTransitivity: calling any unsafe function makes you unsafeFix the called function or protect the call

Reentrant Functions

A function is reentrant if it accesses NO shared variables. Reentrant functions are a proper subset of thread-safe functions. They require no synchronization and are the gold standard for concurrent code. All reentrant functions are thread-safe, but not all thread-safe functions are reentrant.

Thread-Level Parallelism

Using multiple threads to exploit multi-core processors for faster computation.

Parallelizing a Sum

// Each thread sums a portion of the array void *thread_fn(void *arg) { int myid = *((int*)arg); int start = myid * (n / nthreads); int end = start + (n / nthreads); double local_sum = 0; for (i = start; i < end; i++) local_sum += a[i]; psum[myid] = local_sum; // no lock needed - separate array elements }
