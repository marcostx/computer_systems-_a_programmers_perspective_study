Two main tasks of the linker:
1. Symbol Resolution — Associate each symbol reference with exactly one symbol definition from the symbol tables of the input object files.
2. Relocation — Merge separate code/data sections and relocate symbols from relative offsets to final absolute memory addresses.

Symbol Resolution

Symbol Types

SymbolStrengthExamples
FunctionsStrongint main() { ... }
Initialized globalsStrongint x = 10;
Uninitialized globalsWeakint x;

Resolution Rules

Rule 1: Multiple strong symbols with the same name → Linker Error!
Rule 2: One strong symbol + one or more weak symbols → Choose the strong symbol
Rule 3: Multiple weak symbols, no strong → Pick any one (arbitrary, dangerous!)
Warning: Rule 2 and 3 can introduce subtle bugs. If int x; appears in one file and double x = 3.14; in another, the linker silently uses the strong definition. The weak reference may interpret the memory differently (type mismatch). Use -fno-common and static to avoid issues.

Linker Algorithm (Left-to-Right Scan)

The linker maintains three sets:
• E = set of relocatable object files to be merged
• U = set of unresolved symbols
• D = set of defined symbols

For each input file f on the command line (left to right):
1. If f is an .o file: add to E, update U and D
2. If f is an .a archive: scan members; only pull in modules that resolve symbols in U
3. After all files processed: if U is non-empty → linker error
Order matters! gcc main.o -lvector -lm works, but gcc -lvector -lm main.o may fail. Libraries should come after the object files that reference them. For circular dependencies, repeat: gcc main.o -lfoo -lbar -lfoo.

Two Relocation Steps

Step 1 — Relocating sections: Merge all .text sections into one, all .data sections into one, etc. Assign runtime memory addresses.
Step 2 — Relocating symbol references: Use relocation entries (.rel.text, .rel.data) to patch every symbol reference to point to the correct runtime address.
Relocation entry types (x86-64):
• R_X86_64_PC32 — PC-relative reference (32-bit), used for function calls and local data
• R_X86_64_32 — Absolute reference (32-bit), used for absolute addressing

Dynamic Linking

Static vs Dynamic Linking

Static Linking

  • Library code copied into executable
  • Larger executable size
  • No runtime dependencies
  • Must recompile to update library
  • Uses .a archive files

Dynamic Linking

  • Library loaded at runtime
  • Smaller executable size
  • Shared memory across processes
  • Library updates without recompile
  • Uses .so shared object files

GOT and PLT: Lazy Binding

1
Program calls printf() for the first time → jumps to PLT[n] (Procedure Linkage Table entry)
2
PLT[n] does indirect jump through GOT[n] (Global Offset Table). Initially GOT[n] points back to PLT[n]+6
3
PLT[n] pushes relocation index onto stack and jumps to PLT[0]
4
PLT[0] calls the dynamic linker (ld-linux.so), which resolves the symbol
5
Dynamic linker patches GOT[n] with the real address of printf()
6
Subsequent calls to printf(): PLT[n] → GOT[n] → directly to printf (no linker involved)
Key concepts:
• GOT (Global Offset Table) — Data section array. Each entry holds the runtime address of a function/variable. Writable, updated by the dynamic linker.
• PLT (Procedure Linkage Table) — Code section stubs. Each entry is a small trampoline that jumps through the corresponding GOT entry.
• Lazy binding — Symbols are resolved on first use rather than at load time. Controlled by LD_BIND_NOW environment variable.
