Interrupt
Async — caused by events external to the processor.
Examples: Timer interrupt, keyboard I/O, disk controller completion, network packet arrival.
Behavior: Hardware sets the processor's interrupt pin. After current instruction completes, processor reads exception number and calls the interrupt handler. Returns to the next instruction as if nothing happened.

How it works: The CPU uses the exception number as an index into the exception table (base address in a special register). Each entry is the address of the corresponding handler. On an exception, the CPU pushes return address + flags onto the kernel stack, then jumps to the handler.

Return behavior by class:
• Interrupt: iret to next instruction I_next
• Trap: iret to next instruction I_next
• Fault: iret to current instruction I_curr (re-execute) or abort
• Abort: Does not return — terminates the process

The process is either executing on a CPU or waiting to be scheduled (ready). The kernel scheduler decides which running process gets CPU time. A process transitions to running when it is first created (after fork) or when it receives a SIGCONT signal.

fork() Return Values
To parent: Returns the PID of the newly created child process (positive integer).
To child: Returns 0.
On error: Returns -1 (no child created).

Key properties: child gets an identical (but separate) copy of the parent's virtual address space, file descriptors, and signal handlers. Parent and child execute concurrently -- their relative order is nondeterministic.

Key concepts:
• Pending signal: Signal sent but not yet received. At most one pending signal of each type (not queued).
• Blocked signal: Process can block signals using sigprocmask(). Blocked signals remain pending until unblocked.
• signal(signum, handler) or sigaction() — Install a custom signal handler.
• Signals of the same type are not queued: if a second SIGCHLD arrives while the first is pending, it is discarded.

Context Switch Steps

1
Process A is running in user mode. A timer interrupt (or syscall/fault) triggers a transition to kernel mode.
2
Kernel saves context of A — general-purpose registers, PC, kernel stack pointer, %rsp, page table base register, etc.
3
Kernel scheduler decides to run Process B. May be triggered by timer interrupt, I/O completion, or explicit sleep()/read().
4
Kernel restores context of B — loads B's saved registers, switches to B's page table (new virtual address space).
5
Process B resumes execution in user mode from exactly where it left off. It has no idea a switch occurred.
6
Later, another interrupt triggers the reverse: save B, restore A, and Process A continues.
What is a context? The kernel maintains a context for each process: the state needed to restart a preempted process. This includes general-purpose registers, floating-point registers, the program counter, user stack, status registers, kernel stack, and various kernel data structures (page table, process table, file table).

Context switches are triggered by: system calls (e.g., read(), sleep()), interrupts (e.g., timer at ~1-10ms intervals), or faults. The scheduler uses a priority-based algorithm to pick the next process.
